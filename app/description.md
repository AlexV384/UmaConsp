# Документация приложения "Umaconsp" — AI‑чат с поддержкой изображений

## 1. Общее описание

Umaconsp — это Android‑приложение, реализующее чат с искусственным интеллектом. В качестве бэкенда используется локально запущенная модель **Qwen3-VL:4B** через **Ollama**. Приложение поддерживает:

- Текстовые сообщения и отправку изображений.
- Потоковый вывод ответа AI (Server‑Sent Events) с постепенным обновлением интерфейса.
- Отображение **размышлений** (thinking) модели (если они есть).
- Сохранение истории чатов в локальной базе данных (Room).
- Переименование и удаление чатов.
- Экспорт чата в текстовый файл.
- Смену темы (светлая/тёмная) и настройку IP‑адреса сервера.

Приложение построено на **MVVM** с использованием **Jetpack Compose** для UI, **Coroutines + Flow** для асинхронности, **Room** для хранения, **Retrofit + OkHttp** для сети и **DataStore** для настроек.

## 4. Описание ключевых классов

### 4.1. MainActivity
- **Роль**: Главная активность, настройка темы, навигации и DI.
- **Методы**:
    - `onCreate()`: инициализирует `ThemeManager`, `SettingsManager`, `ChatListViewModel`.
    - Настраивает `ModalNavigationDrawer` с боковым меню настроек.
    - Использует `NavHost` для навигации между `ChatListScreen` и `ChatScreen`.
    - Подписывается на удаление чата через `chatListViewModel.chatDeleted` и закрывает текущий экран, если удалён открытый чат.
    - Передаёт `ChatListViewModel` через `CompositionLocalProvider` (см. `LocalChatListViewModel`).

### 4.2. UmaconspApplication
- **Роль**: Singleton‑приложение, инициализирует Room базу данных.
- **Поля**:
    - `instance`: ссылка на экземпляр приложения.
    - `database`: ленивая инициализация `AppDatabase`.
- **Метод** `getAppContext()`: возвращает контекст приложения, используется в ViewModel для получения строк ресурсов.

### 4.3. Пакет data.database
#### AppDatabase
- **Роль**: Абстрактный класс RoomDatabase, определяет DAO и версии БД.
- **Миграции**: `MIGRATION_2_3` добавляет колонку `thinking` в таблицу `messages`.
- **Метод** `getDatabase()`: потокобезопасный синглтон.

#### ChatEntity
- **Поля**: `id` (PK), `title`, `lastMessage`, `lastMessageTime`.
- **Таблица**: `chats`.

#### MessageEntity
- **Поля**: `id` (PK), `chatId` (FK), `text`, `isUser`, `imageUrls` (строка с разделителем `,`), `timestamp`, `thinking`.
- **Таблица**: `messages`.

#### ChatDao
- `getAllChats()`: Flow всех чатов, отсортированных по времени последнего сообщения.
- `getChatById()`, `insertChat`, `updateChat`, `deleteChat`.

#### MessageDao
- `getMessagesForChat()`: Flow сообщений чата, отсортированных по времени.
- `getMessageById()`, `insertMessage`, `updateMessage`, `deleteMessagesForChat`.

### 4.4. Пакет domain.model
#### Message
- **Роль**: Модель данных, используемая в UI.
- **Поля**: `id`, `text`, `isUser`, `imageUrls`, `thinking`.

### 4.5. Пакет presentation.chatlist
#### ChatListViewModel
- **Роль**: Управляет списком чатов.
- **Поля**:
    - `chats: Flow<List<ChatItem>>` — преобразует `ChatEntity` в `ChatItem`.
    - `chatDeleted: SharedFlow<String>` — уведомляет об удалении чата.
- **Методы**:
    - `createNewChat()`: создаёт новый чат с заголовком "Чат N".
    - `addMessage()`: добавляет сообщение в БД и обновляет `lastMessage` чата.
    - `updateMessage()`: обновляет текст и/или размышления сообщения.
    - `updateThinking()`: обновляет поле thinking пользовательского сообщения.
    - `deleteChat()`, `renameChat()`.
    - `getChat()`, `getMessagesFlow()`.

#### ChatListScreen
- **Роль**: UI списка чатов.
- **Компоненты**:
    - `TopAppBar` с кнопкой настроек.
    - `FloatingActionButton` для создания нового чата.
    - `LazyVerticalGrid` с карточками чатов.
    - Диалоги переименования и подтверждения удаления.

### 4.6. Пакет presentation.chat
#### ChatViewModel
- **Роль**: Управляет текущим чатом, отправкой сообщений и потоковым получением ответа.
- **Поля**:
    - `_status`: статус (соединение, генерация, ошибка).
    - `_selectedImageUris`: список URI выбранных изображений.
    - `currentEventSource`: текущее SSE‑подключение.
    - `currentUserMessageId`: ID последнего отправленного сообщения пользователя.
    - `lastThinkingUpdate`, `lastTokenUpdate`: временные метки для throttle.
- **Методы**:
    - `sendMessage()`: отправляет текстовое сообщение или с изображениями. Создаёт сообщение пользователя, очищает выбранные изображения, запускает поток.
    - `sendTextMessageStream()`: формирует `multipart/form-data` запрос с полем `message`, добавляет заголовок `Accept: text/event-stream`.
    - `sendMessageWithImagesStream()`: сжимает изображения (макс. 1024px, качество 80%), сохраняет временные файлы, добавляет их в multipart‑запрос.
    - `startEventSource()`: создаёт `EventSource` через `EventSources.createFactory()`. В `EventSourceListener`:
        - `onOpen()`: создаёт пустое сообщение AI в БД.
        - `onEvent()`: обрабатывает JSON. При получении `thinking` — накапливает и обновляет через throttle (100 мс). При получении `token` — накапливает текст и обновляет через throttle. При ошибке — добавляет сообщение об ошибке.
        - `onClosed()`: финальное обновление сообщения AI, очистка временных файлов.
        - `onFailure()`: обработка сетевых ошибок.
    - `addSelectedImage()`, `removeSelectedImage()`, `clearSelectedImages()`: управление выбранными изображениями.

**Особенности реализации потокового вывода**:
- Используется **throttle** (100 мс) для обновления БД, чтобы не перегружать систему частыми записями.
- Последнее обновление (финальный текст) выполняется в `onClosed()`.
- Благодаря этому UI обновляется постепенно, как и размышления.

#### ChatScreen
- **Роль**: UI экрана чата.
- **Компоненты**:
    - `TopAppBar` с кнопками назад, сохранения, настроек, переименования.
    - `Card` для отображения статуса.
    - `LazyColumn` с сообщениями (`reverseLayout = true` для прокрутки вниз).
    - `LazyRow` для превью выбранных изображений.
    - `InputSection` (поле ввода + кнопка отправки + кнопка выбора изображений).
    - Диалог переименования чата.
- **Логика**:
    - Подписка на `messagesFlow` из `ChatListViewModel` для получения истории.
    - Запрос разрешений на чтение медиа (Android 13+).
    - Использование `rememberLauncherForActivityResult` для выбора нескольких изображений и сохранения файла.
    - Автоматическая прокрутка вниз при появлении новых сообщений.

### 4.7. Пакет presentation.settings
#### SettingsManager
- **Роль**: Хранение IP‑адреса сервера в DataStore.
- **Поля**: `serverIpFlow: Flow<String>`.
- **Метод** `setServerIp()`.

#### ThemeManager
- **Роль**: Хранение выбранной темы (тёмная/светлая) в DataStore.
- **Поля**: `isDarkTheme: Flow<Boolean>`.
- **Метод** `setDarkTheme()`.

#### SettingsDrawerContent
- **Роль**: UI бокового меню.
- Содержит переключатель темы и поле ввода IP‑адреса.

### 4.8. Пакет utils
#### AiApi
- **Роль**: Настройка OkHttp клиента и базового URL для API.
- **Поля**:
    - `currentIp`: изменяемый IP (при изменении сбрасывается `_service`).
    - `client`: OkHttpClient с таймаутами 60 секунд и повторными попытками.
    - `service`: ленивый Retrofit‑сервис (не используется, так как реальные запросы идут через OkHttp + SSE).
    - `baseUrl`: строка `http://$currentIp:5002/`.
- Интерфейс `AiApiService` описан, но не используется (запросы формируются вручную для поддержки SSE).

#### LocalChatListViewModel
- **Роль**: `CompositionLocal` для проброса `ChatListViewModel` в дерево Compose (избегает передачи через параметры).

## 5. Сетевой слой и потоковая передача

### 5.1. Формирование запроса
- Запросы отправляются через **OkHttp** с методом `POST` и заголовком `Accept: text/event-stream`.
- Тело запроса — `multipart/form-data`:
    - Поле `message` — текст.
    - Поле `image` — файлы изображений (после сжатия и временного сохранения).
- Сервер (Python FastAPI) возвращает поток SSE с JSON‑объектами:
    - `{"thinking": "..."}` — размышления модели.
    - `{"token": "..."}` — часть ответа AI.
    - `{"error": "..."}` — ошибка.
    - `{"done": true}` — завершение.

### 5.2. Обработка потока
- `EventSourceListener` принимает строки JSON и разбирает их.
- Для `thinking` обновляется `accumulatedThinking` и вызывается `updateThinking` пользовательского сообщения.
- Для `token` обновляется `accumulatedText` и вызывается `updateMessage` для сообщения AI.
- Все обновления БД выполняются с throttle 100 мс (через сравнение времени последнего вызова). Это снижает нагрузку на БД и сохраняет плавность UI.
- В конце потока выполняется финальное обновление, гарантирующее полный текст.

### 5.3. Управление жизненным циклом
- `currentEventSource` отменяется при очистке ViewModel.
- Если чат был удалён во время генерации, обработка событий прерывается.

## 6. Хранение данных

- **Room** используется для хранения чатов и сообщений.
- Связь между чатами и сообщениями — через `chatId` (внешний ключ с каскадным удалением).
- Поле `thinking` хранится отдельно для каждого сообщения (используется для пользовательских сообщений).
- `imageUrls` хранятся как строка с разделителем `,`.
- Для обновления UI используются `Flow` из DAO, что обеспечивает автоматическую перерисовку Compose при изменениях.

---

## 7. Настройки и тема

- **DataStore Preferences** хранит:
    - `dark_mode` (boolean)
    - `server_ip` (string)
- `ThemeManager` и `SettingsManager` предоставляют `Flow` для наблюдения за изменениями.
- В `MainActivity` изменения IP сразу обновляются в `AiApi.currentIp`.
- Тема применяется через `UmaconspTheme` (кастомная тема Compose, основанная на `MaterialTheme`).

## 8. Возможности пользователя

- **Отправка сообщений**: ввод текста, прикрепление нескольких изображений (сжатие до 1024px).
- **Потоковый ответ**: текст появляется по мере генерации.
- **Размышления**: если модель их предоставляет, они отображаются под сообщением пользователя в раскрывающемся блоке.
- **Сохранение чата**: экспорт в `.txt` файл через системный диалог сохранения.
- **Управление чатами**: переименование (долгий тап по карточке или кнопка в верхней панели), удаление.
- **Настройки**: смена темы, смена IP сервера (по умолчанию `192.168.1.67`).

## 10. Заключение

Приложение демонстрирует современный подход к разработке Android‑приложений с использованием Jetpack Compose, Coroutines и локального AI. Ключевая особенность — плавный потоковый вывод ответа модели, реализованный через SSE и оптимизированные обновления БД. Архитектура MVVM с разделением на слои обеспечивает тестируемость и поддерживаемость кода.
